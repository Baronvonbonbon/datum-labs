# DATUM ↔ DATUM Labs — Structural Gap Findings

**Date:** 2026-05-30
**Author:** review pass cross-referencing the `datum` protocol surface (alpha-5 v5,
53 contracts live on Paseo) against what the `datum-labs` test bench actually
exercises. Complements the labs' own self-analysis in `ROADMAP.md`,
`RELAY-TOOLKIT-SCOPE.md`, and `BRAINSTORM.md` — this doc is the *external* read:
what the protocol can do that the lab has never touched.

## The two trees

- **`datum/` — the protocol.** Alpha-5 v5 on Paseo (2026-05-23). 53 production
  contracts + 5-contract token plane, 1579 tests, browser extension, web app (82
  pages), SDK v3.4, WordPress plugin, Pine light-client RPC, IPFS node, ZK
  circuits. Supports three settlement paths, AssuranceLevel L0–L3, dual-sig, ZK
  (impression / identity / nullifier / interest), DATUM token emissions +
  fee-share + bootstrap, a three-phase governance ladder, and symmetric
  publisher/advertiser/relay fraud tracks.
- **`datum-labs/` — the test bench.** Deliberately outside the protocol git tree.
  Services run *against the live network* to generate real activity and measure
  network effects: `indexer`, `publisher-shortener`, `relay`,
  `advertiser-cosigner`, `deploy` (tunnel + systemd).

## Critical gaps — protocol surface the lab has never touched

Through-line: **every lab test used synthetic, fully-controlled actors**
(`userSig = 0x00`, relay historically held both keys). Ranked:

### 1. Real user / extension path (ROADMAP A2) — the single biggest gap
Nothing has exercised extension → SDK challenge-response → on-device Blake2-256
claim → **user-signed** claim. The "impressions are real, not injected" property
lives entirely here and has never run. The indexer faithfully counts
`ClaimSettled`, but every settled claim came from `inject-claim.mjs` with a null
user sig. This is also `datum`'s own top open item ("E2E browser validation").
**Until one real browser settles one real impression, the mechanism is unproven.**
*(Deferred to manual verification by Kasey.)*

### 2. Only one settlement path is tested
The lab exercises `settleSignedClaims` (dual-sig / L2) exclusively. The L0/L1
zero-gas path real users would actually use (`Relay.settleClaimsFor` with
`userSig` + optional publisher cosig) and the bonded-relay path are untouched.
BRAINSTORM 4.4 (compare all three `data-relay-mode` paths on real gas/latency) is
the missing decision input — and feeds gap #3.

### 3. Settlement aggregation — the unmade architectural call (ROADMAP B1)
Measured cap is **batch = 1, ~49k gas/claim**, ~94k settlements/day per signer;
real ad volume is millions/day and *gas cost itself* is the wall (not nonce
throughput). No Merkle/ZK checkpoint design exists. Decide *before* scaling relay
infra or a submitter pool. **Scoped in `SETTLEMENT-AGGREGATION-DESIGN.md`.**

### 4. Token plane + user payout loop is invisible (ROADMAP C2)
Users "earn" DATUM, but `MintCoordinator → EmissionEngine → MintAuthority →
BootstrapPool → FeeShare → Vesting → Wrapper` is never indexed or validated
end-to-end. The indexer tracks **only 6 events** — no stake, governance,
token-plane, or escrow events. You literally cannot see whether the user
incentive pays out. If withdraw→value doesn't work smoothly, extension adoption
never happens → no impressions → no market.

### 5. ZK / assurance gradient never engaged
The lab relay fills ZK/PoW/nullifier fields with zeros. AssuranceLevel L3
(ZK-only floor), `DatumNullifierRegistry`, `DatumInterestCommitments`, identity
circuits — all dark. With #1, the anti-sybil primitives (ROADMAP B3) are
untested against any adversary.

### 6. Supply is structurally scarce (ROADMAP C1)
Liquidity ratio ~55: hundreds of campaigns, **3 publishers**. The shortener is
*one* publisher app; BRAINSTORM Tier 1 lists 7 archetypes. The existing
WordPress plugin (1.7) and the docs site (1.6) are near-free supply not yet
deployed.

### 7. Indexer can't prove value conservation or CPC
Probe 4.1 (Σ vaults + fees + credits == escrow deducted) needs budget/vault
events it doesn't index. The full payment split *is* in `ClaimSettled`
(`publisherPayment`, `userPayment`, `protocolFee`), but escrow debits and vault
credits/withdrawals are not. Clicks carry no user address (publisher stand-in),
so CTR/attribution (2.3) and k-factor are uncomputable — the indexer README
admits k-factor isn't computed.

### 8. Process tension: production tooling inside a throwaway lab
Production-grade relay tooling is being built inside a self-described disposable
lab. The relay toolkit and indexer both need a **graduation path** (own repo, or
`datum/relay-bot` + webapp-reads-from-indexer) before they accrete more
pre-mainnet weight outside the audited tree.

## Recommended sequencing

1. **(Kasey, manual)** Validate the real extension flow (A2) against the
   publisher-shortener — one real browser → independent dual-sig → on-chain.
   Closes #1 and datum's top open item together.
2. **Exercise the L0/L1 user-signed path** and run BRAINSTORM 4.4 — real
   gas/latency across all three relay modes. Input for #3.
3. **Make the aggregation decision (#3)** — see
   `SETTLEMENT-AGGREGATION-DESIGN.md`. The one call that gates the economics.
4. **Extend the indexer to token-plane + budget/vault events** — make the
   user-payout loop (#4) and value-conservation probe (4.1) measurable.
5. **Add 2 near-free publishers** — SDK in `datum/docs` (1.6) + WordPress demo
   (1.7) — to move the liquidity ratio and prove the non-SDK path.
6. **Finish relay toolkit M2→M3** (`/claim` rate-limit, low-gas alert, dashboard
   auth) and decide the graduation path out of the lab.

Items 3 and 4 are buildable now without touching the protocol contracts or
needing the browser. Item 3 is the one to force a decision on early — it can
invalidate the relay-scaling work the toolkit is otherwise heading toward.
